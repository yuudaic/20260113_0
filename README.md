# READMEの役割（この文書の使い道）
このREADMEは、**すでに生成済みのLPコード（HTML/CSS/JS）を継続的に改善・運用するための判断基準**です。  
コピーや構成の「新規生成」を目的とせず、**改修時に守るべき要件・制約・期待値調整の方針**を明文化します。

---

# LP概要（1〜2行）
地方の企業で働く従業員（業務利用前提）が、生成AIを**安全に試し、業務活用の入口を掴む**ための「生成AI体験セミナー（無料・現地開催）」の参加申込みLP。

---

## 目的（CV・行動目標）
- 主要CV：**セミナー参加申込みの獲得**
- CTAは**2系統併用**
  - 電話（`tel:`でスマホから発信可能）
  - 申込みフォーム（Googleフォーム想定の導線）
- スマホ閲覧で**5分以内**に以下が理解できる状態を作る
  - 誰向けか（業務利用前提の従業員）
  - 何が学べるか（範囲・到達イメージ）
  - 注意点（リスク・前提・限界）
  - 申込み方法（電話／フォーム）

---

## 想定ターゲット
- 地方の企業・団体で働く従業員（業務利用が前提）
- IT・AIに強くない層も含む（初心者を前提）
- 主な不安・悩み（LP内で先回りして言語化）
  - 生成AIが気になるが、業務でどう使えばいいか分からない
  - 情報漏洩や著作権が不安で触れていない
  - ChatGPT / Copilot / Claudeの違いが分からない
  - メール、社内文書、マニュアル作成に時間がかかる
  - 自己流で触ったが、指示の出し方が分からず挫折した

---

## トーン / 世界観 / コピー方針
- 基調：**誠実・安心・過剰に煽らない**
- 伝え方：
  - 不安（情報漏洩・著作権・社内ルール）を**先に解消**してから価値提示
  - 「魔法のツール」ではなく**業務の補助輪**として表現
  - 専門用語は最小限、**結論ファースト＋箇条書き中心**
  - 「無料」「定員」は煽らず、**少人数で体験重視の理由**を添える
- 期待値調整：
  - 「必ず効率化できる」「誰でもすぐ成果」などの**断定・成果保証はしない**
  - 代わりに「学ぶ範囲」「扱う深さ」「参加後に役立てやすくなる可能性」を具体化

---

## 提供価値（成果ではなく「状態」）
LPで約束するのは成果ではなく、参加後に到達しやすい“状態”の提示に限る。
- 生成AIの全体像と「できること／できないこと」を整理できる
- 業務利用時のリスクと注意点（情報漏洩・著作権・社内ルール）を理解できる
- 業務で使える**基本的な指示の型（プロンプト）**を持ち帰れる
- メール・文章・マニュアル作成への活用イメージが持てる
- 社内で説明・共有しやすい知識レベルに近づける

---

## サービス設計（段階構成）
- 体験セミナー（単発／3時間／現地開催）
- 「理解→体験→持ち帰り」を軸にした設計
  - 理解：生成AIの位置づけ、できる／できない
  - 体験：文章生成・アイデア出し・業務文書の下書き
  - 持ち帰り：プロンプトの型、注意ポイント、社内共有しやすい整理

---

## 料金の考え方・見せ方
- 受講料：**無料**
- 無料の理由づけは「導入の第一歩」「体験重視」「少人数でのサポート」等、**誠実な説明**に留める
- 金額で煽らない（「今だけ」「限定でお得」等は避ける）

---

## 競合との差別化ポイント（誠実ベース）
- 業務利用における**リスク（情報漏洩・著作権・社内ルール）を正面から扱う**
- 体験だけで終わらず、**業務で再現しやすい“型（プロンプト）”を持ち帰れる**設計
- ChatGPT / Copilot / Claudeの違いを、初心者が混乱しないよう**整理して提示**
- 定員15名は「少人数で体験重視」のため（煽りではなく設計理由として説明）

---

## CTA要件
- CTA種類（常に両方を提示）
  1. **電話**：スマホタップ可能な `tel:` を実装
  2. **申込みフォーム**：Googleフォームへ遷移（LP内はダミーフォームでも可）
- CTA文言のトーン
  - 押し付けず、行動が明確（例：「無料で体験してみる」「参加を申し込む」「空席を確認する」）
- 配置要件
  - セクション末尾に分散配置
  - 最終セクションは丁寧だが明確に背中を押す締め（ただし煽り禁止）
- スマホでは**スティッキーフッター**に「電話／フォーム」を常設

---

## 実装要件（フロントエンド前提）
※既存コードを壊さず、要件を満たしているかを点検・改修するための基準。
- ファイル分離：`index.html` / `style.css` / `main.js`
- OGPメタタグ：X / LINE / Facebookを想定
  - `og:title` / `og:description` / `og:image`
  - `twitter:card` は `summary_large_image`
- UI機能
  - スティッキーヘッダー（主要アンカーへの導線）
  - スマホ用スティッキーフッター（電話／フォームCTA）
  - スムーズスクロール（アンカー移動）
  - トップへ戻るボタン
  - ハンバーガーメニュー（スマホ）
- 画像
  - Unsplashの実URLを使用（研修・ビジネス・学習・PC作業系）
  - 画像は“雰囲気”よりも“安心感・学習・業務”を優先
- フォーム
  - LP内にHTMLフォームを置く場合、送信はダミーで可
  - 実導線はGoogleフォームを想定（リンクの明確化を優先）

## 推奨セクション構成
ナビ（Sticky Header）のアンカーと一致させ、情報の迷子を防ぐ。
1. Sticky Header（概要／対象／学べること／会場／申込み）
2. Hero（業務利用の悩み代弁＋誠実な一文＋2CTA）
3. このセミナーで得られること（状態ベース）
4. こんな方におすすめ（不安・悩みの言語化）
5. 生成AIの誤解を整理（できる／できない）
6. 安全に使うための注意点（情報漏洩・著作権・社内ルール）
7. カリキュラム紹介（範囲・深さ・体験要素）
8. 開催概要（日時・会場・定員・持ち物・参加条件）
9. 講師プロフィール（事実ベース）
10. FAQ（初心者OK？／会社で使える？／PC必須？ 等）
11. 申込み（電話＋フォーム導線の最終集約）
12. Footer（主催者情報・注意書き）

## カリキュラム（チラシ準拠：範囲の固定）

5. プロンプトの基本（役割・指示・条件指定）

## ガードレール（絶対条件：禁止表現・注意）
- 成果保証・断定表現の禁止（例）
  - 「必ず業務を効率化できる」
  - 「誰でもすぐ成果が出る」
  - 「これだけで解決する」
- “安全”の言い切り禁止
  - 「完全に安全」「絶対に漏れない」等は不可
  - 代替：リスク・前提・社内ルール順守を明記し、注意喚起する
- 過剰な煽りの禁止
  - 緊急性の演出（カウントダウン、過度な限定煽り）は避ける
  - 定員15名は「少人数で体験重視」という設計理由に紐づける
- 実績誇張・権威づけの過剰利用を避ける
  - “すごさ”より“安心して参加できる情報の明確さ”を優先
